BigBrother Agent — How it works, how to integrate, and how to test (simple, clear)

Overview
--------
This repository now contains an "agent engine" that runs uploaded installers inside lightweight Docker sandboxes and produces artifacts (installer copy, logs, metrics, file-diff). The agent is intentionally minimal so another backend can call it (programmatically) and consume results. The agent does NOT require a database to run in the default test mode — it uses an in-memory store for run metadata.

Main components
---------------
- Agent entrypoint (dist/agent.js)
  - Programmatic API: startTestRun({ tenantId, softwareId, idempotencyKey, installerBuffer, ... })
  - Creates a test run record (in-memory) and launches the orchestrator pipeline.
- Orchestrator (dist/services/orchestrator.service.js)
  - Pipeline: PROVISIONING → PRE_METRICS → INSTALLING → POST_METRICS → ANALYZING → COMPLETED
  - Uses services to create/start container, copy installer, execute installer, capture metrics, and analyze filesystem changes.
- Sandbox (sandbox/)
  - Minimal Docker image (sandbox/Dockerfile) with entrypoint that waits for /sandbox/installer.exe and simulates an install.
  - This is intentionally lightweight for testing. Replace with a production sandbox image for real Wine installs.
- Artifacts (storage/artifacts/)
  - Binary files and reports are written here, e.g. storage/artifacts/<tenant>/<run-id>/installer.exe, metrics_report.json, file_diff_report.json, stdout.log, stderr.log.
- In-memory storage shim (dist/lib/prisma.js)
  - Provides a small Prisma-like API so the agent can run without Postgres. Data is ephemeral (lost on restart).

Quick 3-step test (copy-paste)
------------------------------
Run these commands from the project root:

1) Build the sandbox image (only once):
   docker build -t bigbrother-sandbox sandbox

2) Install Node dependencies (only once on this machine):
   npm install

3) Run the test example (runs the agent and a sandbox container):
   node examples/run_test.js

What the test does (short)
-------------------------
- Creates a test run and starts a sandbox container.
- Copies a small dummy PE buffer into the container as installer.exe.
- The sandbox entrypoint waits for the file, then simulates installation and exits.
- The agent captures artifacts and writes them to disk under storage/artifacts/<tenant>/<run-id>/.

How to inspect artifacts safely
-----------------------------
Do NOT execute installer.exe on your host. Use these safe commands (replace <run-id>):
- List files:  ls -la storage/artifacts/tenant-test/<run-id>
- Identify type: file storage/artifacts/tenant-test/<run-id>/installer.exe
- Extract printable strings: strings storage/artifacts/tenant-test/<run-id>/installer.exe | head -n 200
- View first bytes (hex): hexdump -C -n 64 storage/artifacts/tenant-test/<run-id>/installer.exe
- Check SHA-256: shasum -a 256 storage/artifacts/tenant-test/<run-id>/installer.exe
- View JSON reports: jq . storage/artifacts/tenant-test/<run-id>/metrics_report.json
  (If you don’t have jq, use Node: node -e "console.log(JSON.stringify(JSON.parse(require('fs').readFileSync(process.argv[1],'utf8')),null,2))" <file>)

How to integrate this agent with your backend
--------------------------------------------
Two simple options depending on how you want to integrate:

Option A — Direct import (preferred when backend and agent are deployed together)
- Import the agent and call startTestRun programmatically:
  const agent = require('path/to/repo/dist/agent');
  const fs = require('fs').promises;
  const buf = await fs.readFile('/tmp/installer.exe');
  const res = await agent.startTestRun({ tenantId:'t1', softwareId:'s1', installerBuffer: buf, idempotencyKey:'abc' });
- The agent returns immediately with run.id; poll your chosen status endpoint or query the returned run object (if running in-process) to detect completion.
- Artifacts will be written to storage/artifacts/<tenant>/<run-id>/ on disk; your backend can read or move them as needed.

Option B — Queue / worker (recommended for decoupling)
- Have your backend push a job to a queue (SQS/Rabbit/Redis) with installer data or an object-store path.
- Run the agent as a separate worker process that reads the queue, calls startTestRun for each job, and reports back via callback URL or message with the run id and artifact locations.

Notes for integration
- The current agent uses an in-memory store (no Postgres). If you need persistent DB records, restore the Prisma schema and run Postgres, then replace the in-memory shim with the real Prisma client.
- The sandbox image used for testing is a minimal simulator. For real Windows installers you must provide a sandbox image that contains Wine and required tools, and update config.SANDBOX_IMAGE_NAME accordingly.
- By default the agent mounts Docker and uses the Docker daemon. The host running the agent must be allowed to start containers.

Testing checklist (what to verify)
----------------------------------
1. docker images | grep bigbrother-sandbox  → image exists
2. node examples/run_test.js  → script runs until run completes
3. storage/artifacts/<tenant>/<run-id>/ contains:
   - installer.exe (original binary or dummy buffer)
   - metrics_report.json (if metrics captured)
   - file_diff_report.json (filesystem analysis)
   - stdout.log / stderr.log
4. If install fails, inspect container logs: docker ps -a --filter ancestor=bigbrother-sandbox; docker logs <container-id>

Troubleshooting tips (short)
---------------------------
- "Cannot connect to Docker daemon": start Docker Desktop or use Colima (colima start)
- Missing modules errors: run npm install in project root
- Container exited before installer was copied: rebuild sandbox and use the updated entrypoint (it waits up to 30s)

Security notes (short)
----------------------
- Never run untrusted installers on your host machine. The sandbox runs containers but still requires careful host hardening.
- Do not mount sensitive host paths into sandbox containers. The sandbox is for testing and must be run on isolated hosts for untrusted binaries.

"What does it check for?"

Short answer: the agent runs the installer in a Linux sandbox and gathers behavioral + environmental signals. It validates the file looks like a PE, executes it under Wine, and records what changed on the       
  filesystem, what the process did (CPU/memory/network/io), and any logs/exit code. Those signals are saved as artifacts/metadata that your backend/frontend can display.
                                                                                                                                                                                                                     
  Details — what is collected and what the code checks                                                                                                                                                               
  - Installer validation                                                                                                                                                                                             
    - Checks the buffer starts with the PE "MZ" header (basic sanity check that it’s a Windows EXE). See dist/services/installer.service.js:53 (validateExecutable).                                                 
  - Persisting the installer                                
    - Saves the installer as an artifact (local storage) so the backend can reference it later. See dist/lib/prisma.js:65-87 (testArtifact.create) and dist/agent.js where artifacts are saved.
  - Copying into the sandbox
    - Creates a tar archive and copies the .exe into the container at /sandbox/installer.exe. See dist/services/installer.service.js:65-76 (createTarArchive / copyInstallerToContainer).
  - Executing the installer
    - Runs the installer inside the container via Wine, with an Xvfb wrapper; captures stdout/stderr and exit code. See dist/services/installer.service.js:83-96 and the command used at
  dist/services/installer.service.js:88.
    - Timeout handling: installer execution is raced against a timeout to avoid hangs (dist/services/installer.service.js:91-96).
  - Metrics before/after (process & resource activity)
    - Captures Docker stats (CPU, memory, network, block I/O, pid count) before and after install, then computes deltas (e.g., CPU% change, memory change, network bytes). See dist/services/metrics.service.js:46-56
   (normalizeStats), dist/services/metrics.service.js:115-121 (captureSnapshot), and buildReport at dist/services/metrics.service.js:132-137.
    - The orchestrator triggers before/after snapshots; see dist/services/orchestrator.service.js:85 (before snapshot) and dist/services/orchestrator.service.js:99-102 (after + report save).
  - Filesystem analysis (what files changed)
    - Uses docker diff to list added/changed/deleted files and directories touched. See dist/services/file-analysis.service.js:82-88 and 93-95.
    - Flags writes to a configurable list of “banned” or sensitive directories (e.g., /etc/sudoers, ssh keys, cron, init scripts) — touching these is marked suspicious. See the banned list at
  dist/services/file-analysis.service.js:48-62 and violation detection at dist/services/file-analysis.service.js:96-101.
    - Estimates disk footprint of added files (using du inside the container when possible; falls back to a rough estimate). See dist/services/file-analysis.service.js:134-169.
  - Artifacts produced
    - Installer (original .exe)
    - Stdout and stderr logs from installer run
    - Metrics report (before/after snapshots + delta)
    - File-diff report (list of changes, banned violations, directories touched, footprint)
    - These are saved as artifacts and linked into the TestRun object (dist/lib/prisma.js and the orchestrator saving steps — see dist/services/orchestrator.service.js around the save calls).
  - Failure handling / safety
    - Timeouts, Docker errors, or exec errors are caught; the orchestrator marks the run FAILED or TIMED_OUT and still attempts cleanup (container stop/remove). See orchestrator error handling at
  dist/services/orchestrator.service.js:114-134.

  What it does NOT reliably do (limitations)
  - This is a behavioral and heuristic analysis, not a full signature AV engine. It won’t necessarily detect every malicious payload.
  - It relies on Wine/Linux sandbox behavior for .exe execution; some Windows-only behaviors or kernel-level effects won’t show up.
  - False positives/negatives are possible (e.g., legitimate installers may touch system dirs or spawn processes that look suspicious).
  - Architecture/image issues (e.g., incompatible sandbox image) can cause the container to exit before analysis; that’s an environment problem, not a detection rule.

  How the results map to your backend/frontend
  - The agent writes artifact metadata (object key, hash, size) and JSON reports (metrics, file-diff). The backend can ingest those artifacts or read them from ARTIFACT_STORAGE_PATH and present:
    - Installer metadata (hash, size)
    - Installer stdout/stderr and exit code
    - File-diff summary and list of suspicious/banned paths touched
    - Resource delta chart (CPU/memory/network changes)
    - Estimated disk footprint
  - Integration points: after the orchestrator sets COMPLETED, you can POST the TestRun + artifact metadata to your backend (hook point in dist/services/orchestrator.service.js:108-112).
